3.2 Guia de Despliegue (Deployment)

Prerequisitos

  • Docker 27+ y Docker Compose v2
  • Kubernetes 1.30+ (cluster)
  • .NET SDK 8.0
  • Node.js 20.11+ (para MameyNode)
  • Go 1.22 (para WAMPUM)
  • kubectl, helm configurados

Build de Microservicios

# Build individual
cd 08-dotnet/microservices/ServiceName
docker build -t ierahkwa/service-name:v3.3.0 .

# Build todos (script)
for dir in 08-dotnet/microservices/*/; do
  name=$(basename "$dir" | tr '[:upper:]' '[:lower:]')
  docker build -t "ierahkwa/${name}:v3.3.0" "$dir"
done

Deploy a Kubernetes

# Crear namespace
kubectl create namespace ierahkwa

# Crear secrets
kubectl create secret generic ierahkwa-secrets \
  --from-literal=jwt-secret=$JWT_SECRET \
  --from-literal=db-password=$DB_PASSWORD \
  -n ierahkwa

# Aplicar manifests
kubectl apply -f k8s/services/gateway.yml
kubectl apply -f k8s/services/orbital.yml
kubectl apply -f k8s/services/escudo.yml
# ... (todos los NEXUS)

# Verificar
kubectl get pods -n ierahkwa
kubectl get services -n ierahkwa
kubectl get hpa -n ierahkwa

Deploy MameyNode

cd config/mameynode
docker build -t ierahkwa/mameynode:v3.3.0 .
docker run -d --name mameynode \
  -p 5000:5000 -p 9090:9090 \
  -e NODE_ENV=production \
  -e JWT_SECRET=$JWT_SECRET \
  -e DATABASE_URL=$DATABASE_URL \
  ierahkwa/mameynode:v3.3.0

Deploy WAMPUM Blockchain

cd config/wampum
cp .env.example .env
# Editar .env con passwords reales
docker-compose up -d

Metricas Clave a Monitorear

  • Request Rate: rate(http_requests_total[5m]) — Alerta si < 1 req/s por servicio
  • Latency P99: histogram_quantile(0.99, ...) — Alerta si > 500ms
  • Error Rate: Ratio 5xx — Alerta si > 1%
  • CPU Usage: — Alerta si > 80% sostenido por 5 min
  • Memory: — Alerta si > 85% del limite
  • Pod Restarts: — Alerta si > 3 en 15 min

3.4 Gestion de Incidentes

Niveles de Severidad

NivelDescripcionTiempo de RespuestaEjemplo
P0 - CriticoSistema completamente caido< 15 minutosGateway caido, blockchain detenida
P1 - AltoFuncionalidad principal degradada< 30 minutosPagos fallando, auth caido
P2 - MedioFuncionalidad secundaria afectada< 2 horasDashboard lento, busqueda fallando
P3 - BajoIssue menor, workaround disponible< 24 horasTypo en UI, log excesivo

Procedimiento de Incidentes

  1. Deteccion: Via alertas Grafana, monitoreo, o reporte de usuario
  2. Triage: Asignar severidad (P0-P3) e ingeniero responsable
  3. Comunicacion: Notificar canal #incidents en Slack/chat soberano
  4. Investigacion: Revisar logs (Loki), metricas (Prometheus), traces (Jaeger)
  5. Mitigacion: Rollback, scaling, restart segun sea necesario
  6. Resolucion: Fix permanente, tests, deploy
  7. Post-mortem: Documentar causa raiz, acciones preventivas

Comandos Utiles de Emergencia

# Rollback rapido de un deployment
kubectl rollout undo deployment/service-name -n ierahkwa

# Scale up urgente
kubectl scale deployment/service-name --replicas=5 -n ierahkwa

# Ver logs de un pod con error
kubectl logs -f pod/service-name-xxx -n ierahkwa --tail=100

# Restart de todos los pods de un servicio
kubectl rollout restart deployment/service-name -n ierahkwa

# Ver eventos del cluster
kubectl get events -n ierahkwa --sort-by='.lastTimestamp'

4.7 Guias de Estilo y Estandares de Codigo

HTML/CSS (Plataformas)

  • Usar el design system: shared/ierahkwa.css
  • Variables CSS: --bg, --bg2, --gold, --txt, --txt2, --brd, --r
  • Mobile-first responsive (breakpoints: 600px, 900px)
  • SIEMPRE incluir: skip-nav, aria-label, aria-hidden en emojis
  • SIEMPRE incluir: prefers-reduced-motion, focus-visible
  • Vanilla JS solamente (no frameworks)

  • .NET 8 con Alpine Docker images
  • CQRS pattern con MediatR
  • Health endpoint en /health
  • Stats endpoint en /stats
  • Puerto interno siempre 8080
  • Environment: ASPNETCORE_ENVIRONMENT=Production

Dockerfiles

  • Multi-stage build obligatorio (SDK + Runtime)
  • Base: mcr.microsoft.com/dotnet/sdk:8.0-alpine
  • Runtime: mcr.microsoft.com/dotnet/aspnet:8.0-alpine
  • EXPOSE 8080
  • HEALTHCHECK obligatorio con wget
  • NUNCA hardcodear passwords

Git Workflow

  • Branch naming: feature/nexus-nombre-ticket, fix/descripcion
  • Commits: formato convencional feat: descripcion, fix: descripcion
  • PRs: requieren al menos 1 review antes de merge
  • Main branch protegida: no push directo
  • CI debe pasar antes de merge

4.8 Politicas de Seguridad

Reglas de Seguridad Absolutas

  1. Passwords y secrets SOLO en variables de entorno o secret managers. NUNCA en codigo.
  2. Todo el trafico debe ser TLS 1.3 en produccion.
  3. JWT tokens se firman con RS256 (no HS256).
  4. Datos de ciudadanos se encriptan at-rest y in-transit.
  5. Post-quantum crypto (ML-DSA-65, ML-KEM-1024) en todas las capas.
  6. 2FA obligatorio para todos los empleados en todos los sistemas.
  7. Reportar cualquier incidente de seguridad inmediatamente al Security team.
  8. No instalar software no autorizado en equipos de trabajo.
  9. No compartir credenciales, ni siquiera con companeros.
  10. VPN obligatorio para acceso a sistemas internos desde fuera.

4.9 Procedimientos de Trabajo

Flujo de Desarrollo

  1. Ticket: Tomar un ticket del board (asignado o backlog)
  2. Branch: Crear branch desde main
  3. Develop: Implementar con tests unitarios
  4. Test local: Run tests, build Docker, verificar health
  5. PR: Crear Pull Request con descripcion clara
  6. Review: Esperar y responder a code review
  7. CI: Verificar que CI pasa (lint, tests, build, security scan)
  8. Merge: Merge a main despues de aprobacion
  9. Deploy: Automatico via GitOps (ArgoCD) a staging, manual a produccion
  10. Monitor: Verificar en Grafana que no hay regression

Code Review Checklist

  • El codigo resuelve el ticket/issue?
  • Tests unitarios incluidos?
  • No hay secrets o datos sensibles?
  • Sigue los estandares de estilo?
  • Dockerfile actualizado si es necesario?
  • Accesibilidad (aria-label, skip-nav)?
  • Documentacion actualizada?
  • Health endpoint funciona?
  • No hay breaking changes sin migracion?
